3 
4 Author: Marc Gentile (marc.gentile@epfl.ch)

6 @section Description
7 
8 MPF is a simple multiprocessing frawework for executing code on both SMP
9 (Symmetric Multiprocessing) or on serveurs supporting the Message Passing Interface (MPI).
10 
11 The implemented multiprocessing scheme follows a Producer-Consumer model
12 whereby a "manager" process is responsible for producing and dispatching "jobs"
13 to a pool of "worker" processes. The task of each worker is to process the
14 job assigned to it and once done, supply the result to the manager. The manager
15 can process that result and provide the worker with a new job and so on until
16 no work is left.
17 
18 This scheme works well when the same processing must be applied to a great
19 number of files, e.g. thousands of astronomical images to be processed.
20 
21 Every time it is run, MPF will create a new time-stamped run_xxx directory in
22 the base output directory specified in the configuration file [DIR.OUTPUT]
23 section.
24 
25 In MPF, a Job object is created for each file found to process in a base
26 input directory specified by the user. Such a file may be for example an image
27 or a catalog to process. All the files under the base input directory
28 constitute a 'Dataset' in the sense of MPF. A Dataset can be extended to
29 represent a more complex directory structure.
30 
31 As it is, MPF is only a skeleton that implements a default behavior. One has
32 to extend its classes and overrides some of its methods in order to implement
33 concrete tasks, such as deconvolution or denoising in astronomy.
34 
35 One thus has to create a python module based on MPF, i.e. provide classes that
36 inherits from relevant MPF classes and overrides some of their methods.
37 
38 Assuming such a module has been created, it will have to provide:
39 
40  1. A subclass of class Args in order to tell where to find the
41  default configuration file and optionnally support additional command-line
42  arguments or options
43 
44  2. A subclass of class JobProcessor where the following methods must be
45  overridden to implement custom behavior.
46 
47  - process_job(): where the specific code of the task to perform must be
48  invoked from.
49 
50  - process_job_result(): which defines what to do with the results of the
51  job, like creating catalogs or making plots.
52 
53  - create_job(): factory method for creating a job object if the base
54  Job class must be extended. In such a case, the derived
55  Job class must be defined and its constructor invoked there.
56 
57  - Optionally, it may be useful to override the following methods:
58  - preprocess_job() : processing to be done before process_job() is called
59  - postprocess_job(): processing to be done afer process_job() is called
60  - all_jobs_processed(): invoked once all the jobs have been processed.
61 
62 Modifying other methods has to be made with great care.
63 
64 @section code Source code
65 
66 The MPF code is made of the following files:
67 - mpf.cfg: configuration file
68 - mp_SMP.py: main program for the SMP implementation
69 - mp_MPI.py: main program for the MPI implementation
70 - mp_calc_SMP.py: SMP multiprocessing classes
71 - mp_calc_MPI.py: MPI multiprocessing classes
72 - mp_calc.py: common code for mp_SMP.py and mp_MPI.py
73 - mp_args: command-line argument and options
74 - mp_job: parallel job processing
75 - mp_data: data management in the form of datasets
76 - mp_helper: utility functions
77

89

100 @section Execution
101 
102 The general syntax is:
103 
104  - SMP version: <code>mp_SMP.py [options] </code>
105  - MPI version: <code>mp_MPI.py [options] </code>
106 
107 by default, MPF will look for a configuration file named mpfg.cfg in the
108 ./config directory. The location of this file can be changed using the -c and
109 -d options (see below).
110 
111 The supported options are:
112 
113 <ul>
114  <li> -h, --help Display the usage syntax and the list of supported
115  options. </li>
116 
117  <li> -d, --config-dir Directory where to find the configuration file </li>
118 
119  <li> -c, --config-file Name of the configuration file (default: mpfg.cfg) </li>
120 </ul>
121 
122 To run mp_MPI.py, one has to use the mpirun executable as:
123 
124  - <code>mpirun -np N mp_MPI.py [options] </code>
125 
126 where N is the number of processors (or nodes) to use. MPI must has been
127 installed and configured appropriately.
128 
129 Since the manager process of MPF uses one processor for itself, workers will
130 share N-1 processors.
131 
132 For example: <code>mpirun -np 6 mp_MPI -d mydir -c myconfig.cfg</code> will run mpf
133 on 6 processors, 1 for the manager and 5 for the workers. The manager and each
134 of the workers will look for a configuration file: ./mydir/myconfig.cfg.
