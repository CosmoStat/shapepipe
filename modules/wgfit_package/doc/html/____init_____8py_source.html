00006 @section Description
00007 
00008 The @c gfit method uses a forward model fitting algorithm to measure galaxy shapes. 
00009 
00010 An earlier version of gfit, used to participate in the @c GREAT10 galaxy Challenge
00011 (Kitching et al., 2011, Applied Statistics, Kitching et al, 2012, MNRAS), has been described
00012 in Gentile et al, 2012, astro-ph/1211.4847 (http://arxiv.org/abs/1211.4847).
00013 
00014 The present version has been customized to participate in the @c GREAT3 Challenge 
00015 (http://great3.projects.phys.ucl.ac.uk/leaderboard/).
00016 
00017 The following galaxy models are currently supported:
00018   - @c scsersic: an implementation of galaxy model base on a pure disk Sérsic model 
00019               (http://en.wikipedia.org/wiki/Sersic_profile)  
00020   - @c gbdsersic: an implementation of galaxy model based on the combination of an exponential 
00021                Sérsic profile (sersic index 1) to model the galaxy disk and a de Vaucouleur Sérsic 
00022                profile (Sérsic index 4) to model the galaxy bulge. Both the disk and the bulge 
00023                share the same centroid and the same elipticity. The shear estimator is the 
00024                distortion |e| = (a^2 - b^2)/(a^2 + b^2). The underlying implementation is 
00025                done using @c GalSim v1.x, the "modular galaxy image simulation toolkit", see
00026                https://github.com/GalSim-developers/GalSim.
00027   - @c gbdshsersic:
00028   - @c gbdshsersic: an implementation of galaxy model identical to that of @c gbdsersic except that
00029                the reduced shear |g| = (a - b)/(a + b) is used insead of the distortion |e|. 
00030   - @c gcbdsersic: an implementation of galaxy model identical to that of @c gbdsersic but with
00031                    a varying disk Sérsic index.
00032   - @c gccbdsersic: an implementation of galaxy model identical to that of gcbdsersic except that
00033                     the disk and the bulge Sérsic index can also vary.
00034 
00035 Fitting can currently be performed using two minimizers:
00036   - @c scileastsq: based on the <tt>Levenberg-Marquardt</tt> non-linear least-squares 
00037                    implementation, available with the Python SciPy library 
00038                 (http://docs.scipy.org/doc/scipy/reference/generated/scipy.optimize.leastsq.html))
00039   - @c scdmin: a wrapper to SCDM (Simple Coordinate Descent Minimizer, M. Gentile, 2011-2013)
00040                an implementation of the Coordinate Descent algorithm 
00041                (http://en.wikipedia.org/wiki/Coordinate_descent).
00042 
00043 @c gfit is written in @c Python (http://www.python.org/) and takes advantage of the language
00044 flexibility and richness of features. It relies heavily on the @c numpy and @c scipy 
00045 @c Python C/C++/fortran libraries. 
00046 
00047 The software has been designed in such a way that additional galaxy models and minimizers can be 
00048 plugged-in fairly quickly.
00049  
00050 The implementation of this version relies mainly on the following modules: 
00051 
00052 - ../../mpf_package/mpf (Multi-Processing Framework, M. Gentile, 2013): a Python module for executing code on 
00053   both SMP (Symmetric Multiprocessing) or on serveurs supporting the Message Passing Interface 
00054   (MPI). Refer to the documentation of @c mpf module for additional details.
00055 - @c mpfg3 (M. Gentile, 2013): an extension of @c mpf that provides an adaptation layer to access
00056   the @c GREAT3 datasets. Refer to the documentation of @c mpfg3  module for additional details.
00057 - @c multifit (MulTipurpose Fitting framework, M. Gentile, 2013): a python module that offers a 
00058   common fitting interface to different parametric models and fitting algorithms. Refer to the
00059   documentation of @c Multifit for additional details.
00060 - @c scdm (Simple Coordinate Descent Minimizer, M. Gentile, 2011-2013) an implementation of the 
00061   Coordinate Descent algorithm (http://en.wikipedia.org/wiki/Coordinate_descent). 
00062 - Numpy v1.7.x., SciPy v0.9.x and pyfits Python modules

00121 @section Execution
00122 
00123 The general syntax is: 
00124 
00125 - SMP version: @code gfit_SMP.py [gfit options] @endcode 
00126 - MPI version: @code gfit_MPI.py [gfit options] @endcode
00127 
00128 These executables are located in the installation directory under <tt>gfit-0.7.5/gfit</tt>.
00129 By default, gfit will look for a configuration file with default name @c gfit.cfg, located in the
00130 <tt>./gfit/config directory</tt>. These settings can be changed using the @c -c and 
00131 @c -d options (see below).
00132 
00133 The supported @c gfit options are:
00134 
00135 @verbatim
00136  -h, --help          Display the usage syntax and the list of supported options.
00137 
00138  -d, --config-dir    Directory where to find the configuration file (default: ./config)
00139 
00140  -c, --config-file   Name of the configuration file (default: gfit.cfg) 
00141 @endverbatim
00142 
00143 To run @c gfit_MPI.py, one has to use the mpirun executable as:
00144 
00145 @code mpirun -np N gfit_MPI.py [gfit options]@endcode or 
00146 @code mpirun -np N python [python options] gfit_MPI.py [gfit options]@endcode
00147  
00148 where @c N is the number of processors (or nodes) to use. @c MPI must has been 
00149 installed and configured appropriately. 
00150 On a cluster with a queuing system managed by a job scheduler, @c mpirun will have to be integrated 
00151 in a shell script, invoked using a command like @c qsub. 
00152    
00153 Since the manager process of gfit uses one processor for itself, workers will 
00154 share <tt>N-1</tt> processors. 
00155 
00156 For example: @code mpirun -np 6 gfit_MPI -d mydir -c myconfig.cfg @endcode will run @c gfit
00157 on 6 processors, 1 for the manager and 5 for the workers. The manager and each
00158 of the workers will look for a configuration file: <tt>./mydir/myconfig.cfg </tt>.
00159 
00160 @note Before running @c gfit, edit the configuration file to set the @c BASE_INPUT_DIR value, which
00161       should point to the directory where the @c GREAT3 files reside. 
00162 @note In the configuration file, any environment variable prefixed with @c $ such as @c $HOME will
00163       be expanded when read.

00165 @section gfit_config Configuration
00166 
00167 The behaviour of @c gfit can be controlled though various configuration options located in its
00168 configuration file, <tt>gfit.cfg</tt> by default, in the <tt>gfit/config</tt> directory.
00169 
00170 The format of the configuration file is similar to that of MS Windows&trade; .ini text files, with
00171 @c sections embedded within brackets, like <tt>[section]</tt>, each section containing a set of
00172 (@c key, @c value) pairs. The <tt>.ini</tt> format has been extended to support subsections and
00173 section inheritance. Configuration files are managed by the @c sconfig python module, wiich should
00174 be separately installed (see section <b>Prerequisites</b> above).   
00175 
00176 @subsection dirs Directories
00177 
00178 The input, output and corrresponsing sub-directories are specified under the <tt>[DIR]</tt> section.
00179 
00180 @verbatim
00181 # --- This section contains the directories from where gfit will read or store data                                  
00182 [DIR]    
00183 
00184 [DIR.INPUT]                            # base input directories: currently unused 
00185 BASE_INPUT_DIR = ./input               # location of input objects to process     
00186 
00187 [DIR.OUTPUT]                           # base output directories
00188 BASE_OUTPUT_DIR = ./output             # where to store results, etc.
00189 OUTPUT_PLOT_DIR   = plots              # where to save plots
00190 OUTPUT_MAP_DIR    = maps               # where to save maps
00191 OUTPUT_STAT_DIR   = stats              # where to record statistics
00192 OUTPUT_LOG_DIR    = logs               # where to write logs
00193 OUTPUT_RESULT_DIR = results            # where to write results of runs
00194 OUTPUT_ERROR_DIR  = errors             # where to write errors during runs
00195 @endverbatim
00196 
00197 Each time @c gfit is run, a directory of the form: 
00198 <tt>run_DD.MM.YY_HH.MM.SS.branch.obs_type.data_type</TT> is created under the output directory
00199 specified by the key @c BASE_OUTPUT_DIR under the section @c DIR.DIR_OUPUT. The @c branch, 
00200 @c obs_type and @ data_type components are named after the first capitalized letter of the branch,
00201 data type and observation types respectively. 
00202 
00203 For instance a run in the control-space-variable 
00204 branch will have its data contained in a directory of the form <tt>run_DD.MM.YY_HH.MM.SS.CSV</TT>. 
00205 Or for a run in both the variable_psf-space constant and the full-ground-variable branches,
00206 a directory like <tt>run_DD.MM.YY_HH.MM.SS.VSC_FGV</TT> will be created. The created directory
00207 will in turn contains the sub-directories @c plots, @c maps, @c stats @c logs, @c results and
00208 @c errors. The @c plots, @c maps, @c stats and @c results will have a directory tree-like structure
00209 that mimics the structure of the dataset.
00210 
00211 The <tt>DIR.INPUT</tt> section is currently unused.

00213 @subsection datasets Datasets
00214 
00215 The @c [DATASET] section specifies where input images, catalogs an other requred files to process 
00216 are located. It also specifies Unix-like file matching patterns used to identify those files. 
00217 At a minimum, a @c dataset has a name, a base directory and a a list of file matching patterns. 
00218 
00219 In this version @c gfit uses 3 datasets: one to access the @c GREAT3 images, catalogs and other 
00220 files, @c SExtractor galaxy and PSF catalog files and a @c PSF dataset with PSF images at the 
00221 positions of galaxies. These datasets are described in more detail below.
00222 
00223 @subsubsection g3_dataset GREAT3 dataset
00224 
00225 The @c GREAT3 dataset is identified by the <tt>DATASET</tt> section:
00226 
00227 @verbatim
00228 # --- Main source dataset, in this case GREAT3
00229 [DATASET]                        # note: cannot process both deep and normal images in the same run
00230 
00231 NAME = great3                    # name of dataset to use
00232 BASE_DIR = $HOME/GREAT3          # dataset base input directory
00233 
00234 # --- gfit needs galaxy images and catalogs
00235 FILE_PATTERNS = ["image*.fits", "galaxy_catalog*"]  
00236                                                     
00237 # Note: to process deep images, use: FILE_PATTERNS = ["deep_image*", "deep_galaxy_catalog*"]
00238  
00239 # --- Search criteria, branches: ["control", "real_galaxy", "variable_psf", "multiepoch", "full"]
00240 BRANCHES   = ["control"]               
00241 
00242 # --- Observation type: ["ground", "space"]
00243 OBS_TYPES  = ["ground"]             
00244                            
00245 # --- Data type: ["constant", "variable"]                                           
00246 DATA_TYPES = ["constant"]  
00247 
00248 # --- Set of Image numbers (use [] to process all images (IMAGE_LIST has priority over IMAGE_RANGE) 
00249 IMAGE_LIST  = [0]                   
00250                                     
00251 # --- [Min,Max] image number range (-1 or [] to disable criterion)       
00252 IMAGE_RANGE = []    
00253 
00254 # --- Actual Galaxy stamp size to use for shape measurement
00255 GALAXY_EFFECTIVE_STAMP_SIZE = {"control":{"ground":26, "space":54},\
00256                                 "multiepoch":{"ground":44, "space":44},\
00257                                 "variable_psf":{"ground":44, "space":88},\
00258                                 "real_galaxy":{"ground":44, "space":88},\
00259                                 "full":{"ground":44, "space":88}\
00260                               }          
00261 
00262 # --- Actual PSF stamp size to use for shape measurement
00263 PSF_EFFECTIVE_STAMP_SIZE = {"control":{"ground":26, "space":54},\
00264                              "multiepoch":{"ground":44, "space":44},\
00265                              "variable_psf":{"ground":44, "space":88},\
00266                              "real_galaxy":{"ground":44, "space":88},\
00267                              "full":{"ground":44, "space":88}\
00268                            }  
00269 @endverbatim
00270 
00271 The <tt>BASE_DIR</tt> key specifies where the base @c GREAT3 directories are located, i.e. where
00272 the branch directories (@c control, @c real_galaxy, @c multiepoch, etc.) are to be read from. 
00273 
00274 The <tt>FILE_PATTERNS</tt> key contains a list of pattern matching strings, used to identify which
00275 files are collectively needed to perform shape measurement. A set of such files is processed by 
00276 @c gfit as a @c job. 
00277 For instance, in the context of @c GREAT3, a job  will contain all files related to image @c 000 
00278 in the <tt>control-ground-constant</tt> branch tree: <tt>image-000.0.fits</tt> and
00279 <tt>galaxy_catalog-000.fits</tt>. On a multiprocessor machine or cluster, each job will be assigned 
00280 to one processor.
00281 
00282 The <tt>BRANCHES</tt>, <tt>OBS_TYPE</tt> and <tt>DATA_TYPE</tt> keys respectively specify the lists
00283 of branches (e.g. "control"), observation types (e.g. "ground", "space") and data types (e.g. 
00284 "constant", "variable") to process.  
00285 
00286 The <tt>IMAGE_LIST</tt> key is a list of image numbers to process, like for instance  
00287 <tt>[0, 10, 67]</tt>. 
00288 An empty list <tt>[]</tt> indicates that @e all images in the underlying branch, observation type 
00289 and data type have to be processed.
00290 
00291 The <tt>IMAGE_RANGE</tt> key specifies a range of image numbers to process. for instance, [1, 50] 
00292 will have @c gfit access all images from 1 to 50, 1 and 50 included. Note that the 
00293 <tt>IMAGE_LIST</tt> has priority other <tt>IMAGE_RANGE</tt>: the image range will be ignored unless
00294 <tt>IMAGE_RANGE=[]</tt>.
00295 
00296 The <tt>GALAXY_EFFECTIVE_STAMP_SIZE</tt> and <tt>PSF_EFFECTIVE_STAMP_SIZE</tt> keys respectively 
00297 control the sizes of the postage stamps to cut-out from galaxy and PSF images. @c gfit will 
00298 estimate the centroid of the postage stamp image and cut-out a stamp with the specified size.
00299 Note that since most centroids are shifted by 1 or more pixels, cutting-out a square stamp of the 
00300 exact size specified in <tt>GALAXY_EFFECTIVE_STAMP_SIZE</tt> and <tt>PSF_EFFECTIVE_STAMP_SIZE</tt> 
00301 may not be possible. In such cases, @c gfit will automatically reduce the size of the stamp to the
00302 maximum possible size. A warning will be issued in the log file for that image.

00410 @subsection shape Shape measurement
00411 
00412 The <tt>[SHAPE_MEASUREMENT]</tt> controls the shape measurement operation.
00413 
00414 @verbatim
00415 [SHAPE_MEASUREMENT] 
00416 NORMALIZE_PSF = True                   # if True, normalize the PSF image before convolution
00417 SUBSTRACT_GALAXY_SKY = False           # if True estimate and remove the sky noise in galaxy images
00418 SUBSTRACT_PSF_SKY = False              # if True estimate and remove the sky noise in PSF images
00419 FAILED_ELLIPTICITY_VALUE = 10.0        # if -1,.0 take the value of the fit, even though it failed
00420 GALAXY__SUBSCALE = 1                   # resolution of constructed galaxy models
00421 @endverbatim
00422 
00423 The key <tt>FAILED_ELLIPTICITY_VALUE</tt> sets the value of ellipticity used to flag wrong fits in
00424 the results submission files produced for each processed image. The @c GREAT3 presubmission script 
00425 will ignore entries with ellipticity values greater than or equal to @c 10. If set to @c -1, 
00426 @c gfit will not flag entries with bad fits. This is not recommended.
00427 
00428 Note that @c gfit also always produce, for each image, a result submission file that 
00429 includes @e all fits either good or bad. So the preferred approach is to choose
00430 <tt>FAILED_ELLIPTICITY_VALUE = 10.0</tt>.
00431 
00432 The results files for submission are stored in the directory specified in the 
00433 <tt>OUTPUT_RESULT_DIR</tt> key in the <tt>[DIR.OUTPUT]</tt> section. Their names are of the form: 
00434 <tt>"results_branch_obs_type_data_type-img_no.txt"</tt>, for instance 
00435 <tt>"results_control_ground_constant-000.txt"</tt>. The "full" results file containing all
00436 the fits is prefixed by <tt>full</tt>, like 
00437 <tt>"full_results_control_ground_constant-000.txt"</tt>.
00438 
00439 Lastly, @c gfit also produces for each processed image a files containing only the bad fits. These
00440 files are stored in the directory pointed by the <tt>OUTPUT_ERROR_DIR</tt> key in the 
00441 <tt>[DIR.OUTPUT]</tt> section.

00443 @subsection denoising Denoising
00444 
00445 The <tt>[DENOISING]</tt> section is intended to include any denoising scheme that may be applied 
00446 to each postage stamp. 
00447 
00448 In this version, @c gfit only supports the DWT Wiener wavelet-based denoising algorithm from
00449 Nurbaeva et al., 2011, whose settings can be set in the <tt>[DENOISING.DWTWIENER]</tt> section.
00450 
00451 @verbatim
00452 [DENOISING]
00453 
00454 [DENOISING.DWTWIENER]                  # Denoising with dwtwiener (Nurbaeva et al., A&A, 2011)
00455

00494 @subsection filtering Object Filtering
00495 
00496 This section provides a way to filter-out of "clean" unwanted objects, i.e.
00497 galaxies or stars. A filtering criterion takes the form of a range of allowed values, like 
00498 <tt>[0, 600]</tt>. The value @c None denotes that there is no lower or upper bound. For instance,
00499 <tt>[0, None]</tt> allows for values greater than zero, with no upper limit.
00500 
00501 Filtering takes place <em>before</em> fitting ("pre-filtering") or <em>after</em> 
00502 ("post-filtering"). 
00503 - In this version, pre-filtering is applied on selected SExtractor catalog keywords. The
00504 name of the criterion is the name of the SExtractor keyword prefixed by @c "SE_", like 
00505 @c SE_FLUX_RADIUS.
00506 - Post-filtering can currently be done on the sum of the residuals between observed and built model, 
00507 normalized by the total number of pixels in the postage stamp used for fitting.
00508 
00509 The keyword @c FILTERING_POLICY specifies what should be done with the "filtered" objects. In this 
00510 version, objects are effectively "removed" from the final object catalog 
00511 (<tt>FILTERING_POLICY = remove</tt>). A future version may allow to ascribe a weighting factor
00512 to filtered objects instead of removing them.
00513 
00514 Filtering can be enabled or disabled by specifying <tt>ENABLE_FILTERING=True</tt> or 
00515 <tt>ENABLE_FILTERING=False</tt>. If filtering is enabled, a catalog with un-filtered objects will 
00516 still be produced in order to better investigate the effect of filtering.
00517 
00518 @verbatim
00519 [OBJECT_FILTERING]
00520 
00521 # --- Galaxy ---
00522 [OBJECT_FILTERING.GALAXY]
00523 
00524 ENABLE_FILTERING = True    # set to True to enable galaxy filtering
00525 FILTERING_POLICY = remove  # how to handle filtered data, among: [remove, weight]
00526 
00527 FILTER = {"SE_HLR" : [0, None],\     # allowed SExtractor FLUX_RADIUS range
00528           "SE_FWHM_IMAGE"  : [0, None],\     # allowed SExtractor FWHM_IMAGE range
00529           "SE_FLUX"        : [0, None],\     # allowed SExtractor FLUX_BEST range
00530           "SE_MAG"         : [None, None],\  # allowed SExtractor MAG_BEST range
00531           "SE_SNR"         : [0, None],\     # allowed SExtractor SNR (FLUX_BEST/FLUXERR_BEST)
00532           "NORM_RESIDUALS" : [0, None] \     # allowed range of normalized sum of residuals
00533          }         
00534 
00535 # --- PSF ---
00536 [OBJECT_FILTERING.PSF]
00537 
00538 ENABLE_FILTERING = False   # set to True to enable PSF filtering
00539 FILTERING_POLICY = remove  # how to handle filtered data, among: [remove, weight]
00540

00585 CREATE_CONVOLVED_GALAXY_MOSAIC = True   # if True, create a convolved stamp mosaic

00600 The number of postage stamps to fit within an image is set by the <tt>MAX_NB_STAMPS</tt>. A value
00601 of @c -1 means that @e all postage stamps will be processed. otherwise, @c gfit will process the
00602 @e first number of stamps specified in <tt>MAX_NB_STAMPS</tt>. The sorting order is y then x, so 
00603 @c gfit will proceed row after row in an image, processing each column of a given row.  
00604 
00605 The @c EXTRACT_GALAXY_STAMPS, @c EXTRACT_CONVOLVED_GALAXY_STAMPS, @c EXTRACT_GALAXY_RESIDUALS, 
00606 @c EXTRACT_PSF_STAMPS boolean flags tell whether postage stamps should be extracted as .FITS
00607 files for observed galaxies, convolved galaxies, galaxy residuals and observed PSFs, in this order.
00608 The target directory is that specified as <tt>OUTPUT_RESULT_DIR</tt> in section 
00609 <tt>[DIR.OUTPUT]</tt>.

00612 best-fitted convolved galaxy models inside postage stamps of size set by @c MOSAIC_STAMP_SIZE. The
00613 coordinate of galaxies within the mosaic will be the same as in the source image and catalog. Failed
00614 fits are indicated by a postage stamp with zero pixel values at the corresponding coordinates.<br>
00615 Similarly, if @c CREATE_RESIDUALS_GALAXY_MOSAIC is set to @c True, a mosaic of the residuals 
00616 postage stamps will also be created, where residual postage stamps are the difference between 
00617 observed galaxies and model-convolved galaxies.<br>
00618 Note that, in practice, most postage stamps will have an effective size smaller than the specified mosaic
00619 stamp size (e.g. 44 instead of <tt>MOSAIC_STAMP_SIZE = 48) </tt>. Such stamps will be aligned on 
00620 the bottom-left corner of the mosaic stamp.
00621 
00622 The @c PLOT_GALAXY_STAMPS, @c PLOT_CONVOLVED_GALAXY_STAMPS, @c PLOT_GALAXY_RESIDUALS and

00624 convolved galaxies, galaxy residuals and observed PSFs, in this order.

00626 <tt>[DIR.OUTPUT]</tt>. If @c PLOT_ENABLE_3D is set to @c True, three-dimensional plots are also 
00627 drawn.
00628 
00629 The @c RECORD_FAILED_FITS flag indicates if postrage stamps for which the fit failed should be
00630 stored for later investigation in the directory pointed by <tt>OUTPUT_ERROR_DIR</tt> in section 
00631 <tt>[DIR.OUTPUT]</tt>.
00632 
00633 @c PROGRESS_COUNT specifies how often @c gfit should display or log output information. For 
00634 instance, a value of @c 10 will output every time @c 10 postage stamp images have been fitted.

00647 @section multifit_config Multifit configuration
00648 
00649 @c gfit interacts with the Multifit module through a set of configuration files. There are 
00650 configuration files for fitting methods and for parametric models.
00651 
00652 @subsection model_config Galaxy Model configuration
00653 
00654 A Multifit configuration file is present for each supported galaxy model and bear the same name as 
00655 that of the model. Thus, there are @c 3 model configuration files in this version of @c gfit:
00656 - <tt>scsersic.cfg</tt> for the single Sérsic component model
00657 - <tt>gbdsersic.cfg</tt> for the bulge+disk model
00658 - <tt>gcbdsersic.cfg</tt> for the "complex" bulge+disk model
00659 
00660 For each galaxy model, the list of model parameters is given, specifing the name of each parameter,
00661 its guess value and its bounds (min and max alowed values). 
00662 
00663 @subsection method_config Fitting Method configuration
00664 
00665 A Multifit configuration file is also present for each supported fitting method and bear the same
00666 name as that of the method. Since @c 2 minimizers are supported in this version, there are also 
00667 @c 2 configuration files:
00668 - <tt>scdmin.cfg</tt> for the @c SCDM minimizer
00669 - <tt>scileastsq.cfg</tt> for the <tt>Levenberg-Marquardt</tt> non-linear least-squares 
00670   implementation available with @c SciPy.  
00671 
00672 Sometimes, it is mecessary to provide fitting information that is common to both the model and
00673 the method. This is the case for the @c SCDM minimizer, which uses such information for optimizing
00674 the fitting process. For instance one can specify the initial step value to fit a given parameter.
00675 Such extra fitting information can be kept in a file prefixed by the minimizer name and suffixed by
00676 @c "_fitting". In the case of SCDM, known as @c scdmin by @c gfit, the fitting configuration file 
00677 is  <tt>scdmin_fitting.cfg</tt>. It contains extra fitting information for each galaxy model.
