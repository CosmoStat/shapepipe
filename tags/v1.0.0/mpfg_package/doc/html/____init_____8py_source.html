00006 @section Description
00007 
00008 MPF is a simple multiprocessing frawework for executing code on both SMP 
00009 (Symmetric Multiprocessing) or on serveurs supporting the Message Passing Interface (MPI).
00010 
00011 The implemented multiprocessing scheme follows a Producer-Consumer model 
00012 whereby a "manager" process is responsible for producing and dispatching "jobs"
00013 to a pool of "worker" processes. The task of each worker is to process the 
00014 job assigned to it and once done, supply the result to the manager. The manager
00015 can process that result and provide the worker with a new job and so on until 
00016 no work is left.
00017 
00018 This scheme works well when the same processing must be applied to a great 
00019 number of files, e.g. thousands of astronomical images to be processed.
00020 
00021 Every time it is run, MPF will create a new time-stamped run_xxx directory in
00022 the base output directory specified in the configuration file [DIR.OUTPUT] 
00023 section.  
00024 
00025 In MPF, a Job object is created for each file found to process in a base
00026 input directory specified by the user. Such a file may be for example an image
00027 or a catalog to process. All the files under the base input directory 
00028 constitute a 'Dataset' in the sense of MPF. A Dataset can be extended to 
00029 represent a more complex directory structure.
00030 
00031 As it is, MPF is only a skeleton that implements a default behavior. One has
00032 to extend its classes and overrides some of its methods in order to implement 
00033 concrete tasks, such as deconvolution or denoising in astronomy. 
00034 
00035 One thus has to create a python module based on MPF, i.e. provide classes that 
00036 inherits from relevant MPF classes and overrides some of their methods.
00037 
00038 Assuming such a module has been created, it will have to provide:
00039 
00040   1. A subclass of class Args in order to tell where to find the 
00041    default configuration file and optionnally support additional command-line 
00042    arguments or options
00043 
00044   2. A subclass of class JobProcessor where the following methods must be 
00045      overridden to implement custom behavior.
00046   
00047    - process_job(): where the specific code of the task to perform must be
00048                     invoked from.
00049 
00050    - process_job_result(): which defines what to do with the results of the 
00051                            job, like creating catalogs or making plots. 
00052 
00053    - create_job(): factory method for creating a job object if the base 
00054                    Job class must be extended. In such a case, the derived
00055                    Job class must be defined and its constructor invoked there.
00056 
00057    - Optionally, it may be useful to override the following methods:
00058      - preprocess_job() : processing to be done before process_job() is called   
00059      - postprocess_job(): processing to be done afer process_job() is called   
00060      - all_jobs_processed(): invoked once all the jobs have been processed.
00061 
00062 Modifying other methods has to be made with great care.   
00063

00100 @section Execution
00101 
00102 The general syntax is: 
00103 
00104   - SMP version: <code>mp_SMP.py [options] </code>  
00105   - MPI version: <code>mp_MPI.py [options]  </code> 
00106 
00107 by default, MPF will look for a configuration file named mpfg.cfg in the 
00108 ./config directory. The location of this file can be changed using the -c and 
00109 -d options (see below).
00110 
00111 The supported options are:
00112 
00113 <ul>
00114   <li> -h, --help     Display the usage syntax and the list of supported
00115                       options.  </li>
00116 
00117   <li> -d, --config-dir    Directory where to find the configuration file </li>
00118 
00119   <li> -c, --config-file   Name of the configuration file (default: mpfg.cfg) </li> 
00120 </ul>
00121 
00122 To run mp_MPI.py, one has to use the mpirun executable as:
00123 
00124    - <code>mpirun -np N mp_MPI.py [options] </code>
00125 
00126 where N is the number of processors (or nodes) to use. MPI must has been 
00127 installed and configured appropriately.
00128    
00129 Since the manager process of MPF uses one processor for itself, workers will 
00130 share N-1 processors. 
00131 
00132 For example: <code>mpirun -np 6 mp_MPI -d mydir -c myconfig.cfg</code> will run mpf
00133 on 6 processors, 1 for the manager and 5 for the workers. The manager and each
00134 of the workers will look for a configuration file: ./mydir/myconfig.cfg.
